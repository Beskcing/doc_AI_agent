## 审查规则

### 1. OCR 残留错误
识别 MinerU/OCR 解析残留的识别错误：
- **形近字混淆**：已/己/巳、末/未、人/入、干/千、曰/日、土/士 等
- **数字字母错误**：0/O、1/l/I、2/Z、5/S、8/B 等 OCR 常见混淆
- **化学符号/单位符号**：如 mg→mq、μg→ug→μ9 等
- **乱码字符**：非 ASCII 可打印字符的异常 Unicode 码位
- **中英文标点混用**：如英文逗号出现在中文句子中

### 2. 语义错误
检查内容逻辑和语义正确性：
- **专业术语错误**：国标术语是否被 OCR 改写成无意义词汇（如"试样"→"试祥"）
- **逻辑矛盾**：数值范围是否合理（如 0.1~0.5 变成 0.1~5.0）
- **重复段落**：是否有相同的句子或段落重复出现
- **内容缺失**：是否有明显的句子截断或不完整

### 3. 文字错误
纯粹的文字层面检查：
- **错别字**：同音错字、形近错字
- **漏字/多字**：句子中是否缺字或多出无关字符
- **标点错误**：中文语境下使用了英文标点、标点缺失、多余标点
- **全角/半角混用**：同一个文档中数字、字母的全半角不一致

### 4. 结构完整性
检查文档结构是否合理：
- **标题层级**：章节编号是否连续、层级是否正确（如 3.1 后不应直接跳到 3.3）
- **条款编号**：国标条款编号格式是否正确（如 4.1.1、5.2.3）
- **段落完整性**：是否有只含编号但内容为空的段落

### 5. 格式一致性
检查格式是否统一：
- **单位符号**：同一物理量是否使用统一的单位符号（如 kg 和 千克 不应混用）
- **日期格式**：全文日期格式是否一致（如 2024-01-01 和 2024年1月1日 不应混用）
- **数字格式**：小数位数、千分位分隔符是否一致
- **空格使用**：中文与数字/英文之间是否恰当留空

## 输出格式

请输出严格的 JSON，格式如下：

```json
{
  "issues": [
    {
      "type": "ocr",
      "severity": "high",
      "location": "第3段 / 3.1 条款",
      "original": "试样应在试验前进行预处理。",
      "suggested": "试样应在试验前进行预处理。",
      "reason": "请注意这段文字，确认是否有错误"
    }
  ],
  "summary": {
    "total_issues": 0,
    "ocr_errors": 0,
    "semantic_errors": 0,
    "text_errors": 0,
    "structure_issues": 0,
    "format_issues": 0,
    "overall_quality": "good"
  }
}
```

**字段说明：**
- `type`: 错误类型 — `ocr` / `semantic` / `text` / `structure` / `format`
- `severity`: 严重程度 — `high`(影响理解) / `medium`(可能错误) / `low`(格式瑕疵)
- `location`: 错误所在位置，用段落编号或章节引用描述
- `original`: 原文中实际出现的文本
- `suggested`: 建议修正后的文本
- `reason`: 说明为什么认为这是错误（简要说明推理依据）
- `overall_quality`: 整体质量 — `good` / `fair` / `poor`

**重要约束：**
1. 只对明确有问题的内容提出 issue，不确定的不要报告
2. 不要对正确的专业术语和格式提出"建议优化"
3. 每个 issue 的 `original` 必须是文本中实际存在的原句
4. 如果文本内容正确无误，返回空的 issues 数组，overall_quality 为 "good"
5. 只输出纯 JSON，不要添加任何解释性文字或代码块标记
